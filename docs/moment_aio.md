# **1. Executive Summary**

MOM AIO(Agentic-Interoperable Oracle)는 UMA 옵티미스틱 오라클 모델을 기반으로,
Evidence(증거) 필수 첨부와 멀티 LLM 자동 검증을 결합한 차세대 하이브리드 오라클
프로토콜이다.

기존 오라클의 한계(Chainlink는 판단 데이터 처리 불가, UMA는 고래 조작과 근거
부재)를 해결하면서, 한국 규제 환경에 최적화된 '사실 확인 인프라'로 설계되었다.

### **핵심 공식**

MOM AIO = UMA(유저 직접 제출) + Evidence Lite(증거 박제) + Multi-LLM(AI 자동
검증)

### **핵심 차별화**

유저가 오라클 제출 시 반드시 언론사 기사 등 증거를 첨부해야 하며, 복수의 LLM이
해당 증거가 클레임을 실제로 뒷받침하는지 자동으로 검증한다. 모든 LLM의 모델 ID,
프롬프트 버전, 입출력 해시가 온체인에 투명하게 기록되어 사후 100% 감사가
가능하다.

### **법적 포지셔닝**

MOM AIO는 예측시장(prediction market)이 아닌, '이미 발생한 사실을 확정하는
데이터 인프라'이다. 오라클 보증금은 베팅이 아닌 정직 담보이며, 보상은 소액
정액이다. AP통신, AccuWeather, Sportradar와 동일한 데이터 서비스 카테고리에
해당한다.

# **2. 문제 정의 - 기존 오라클의 한계**

## **2.1 Chainlink의 한계**

Chainlink는 수치/가격 데이터 자동화에는 탁월하나, 'BTS가 공식 컴백했는가?'와
같은 판단형 데이터는 처리할 수 없다. 또한 한국 데이터(멜론 차트, KBO, 한국
정치)를 지원하지 않으며, 노드 운영자가 글로벌 기업(Deutsche Telekom 등) 위주라
한국 로컬 데이터에 인센티브가 없다.

## **2.2 UMA의 한계**

UMA 옵티미스틱 오라클은 어떤 사실이든 검증할 수 있는 유연성이 있으나, 구조적
취약점이 심각하다.

| **문제**       | **내용**                                                                             | **심각도** |
| -------------- | ------------------------------------------------------------------------------------ | ---------- |
| 고래 투표 조작 | 소수 대형 UMA 토큰 홀더가 DVM 투표 결과를 좌우. 2025년 우크라이나 사건에서 실제 발생 | HIGH       |
| 보증금 불균형  | $750 보증금으로 수백만 달러 규모의 결과 조작 시도 가능                               | HIGH       |
| 근거 부재      | Proposer가 증거 없이 결과만 제출. Discord 토론이 유일한 검증 채널                    | HIGH       |
| 한국 맥락 부재 | UMA 토큰 홀더가 K-pop/한국 정치 맥락을 이해하지 못함                                 | HIGH       |
| 사후 감사 불가 | 온체인에 근거 기록이 없어 '왜 그런 판정?' 검증 어려움                                | MEDIUM     |
| 탈중앙화 후퇴  | 2025년 8월 MOOV2로 화이트리스트 도입, 사실상 부분 중앙화                             | MEDIUM     |

## **2.3 MOM AIO가 해결하는 것**

MOM AIO는 UMA의 유연한 유저 제출 모델을 유지하면서, Evidence Lite(증거 박제)와
Multi-LLM(AI 자동 검증)을 추가하여 위 문제를 구조적으로 해결한다.

# **3. 오라클 vs 예측시장 - 법적 경계**

이 구분은 한국 규제에서 가장 중요한 부분이다. MOM AIO가 사행행위법 적용 대상이
아닌 이유를 명확히 한다.

## **3.1 근본적 차이**

| **구분**  | **오라클 (MOM AIO)**            | **예측시장 (Polymarket)**       |
| --------- | ------------------------------- | ------------------------------- |
| 시간축    | 이미 발생한 사실을 확정         | 아직 일어나지 않은 미래에 베팅  |
| 돈의 성격 | 보증금 (정직 담보)              | 베팅금 (투기)                   |
| 수익 구조 | 소액 정액 보상 ($2~5)           | 배당 × 베팅액 (수백~수천% 가능) |
| 손실 구조 | 거짓 보고 시에만 보증금 몰수    | 틀리면 베팅금 전액 손실         |
| 우연성    | 없음 (사실은 이미 확정)         | 있음 (미래는 불확실)            |
| 법적 성격 | 사실 확인 서비스 (데이터업)     | 사행행위 (도박)                 |
| 유사 업종 | AP통신, AccuWeather, Sportradar | 카지노, 스포츠 토토             |

## **3.2 MOM AIO의 법적 방어 논리**

**1. 시간: **이미 발생한 사실만 다룬다. 이벤트 종료 전 제출 불가 (스마트
컨트랙트에서 expiryDate 체크).

**2. 돈: **보증금(정직 담보)이지 베팅이 아니다. 맞으면 돌려받고 소액 보상만
수령.

**3. 목적: **사실 확인 인프라이다. 도박장이 아니라 데이터 서비스.

**4. 증거: **모든 판정에 언론사 기사 근거 필수. 우연성 없음.

**5. AI 검증: **LLM이 근거 유효성 자동 확인. 조작 방지 시스템.

## **3.3 생태계 포지셔닝**

MOM AIO는 '수도관'이다. 수도관에서 물을 받아 사용하는 서비스(예측시장,
파라메트릭 보험 등)가 어떤 사업을 하든, 수도관 자체는 그 사업의 법적 책임을 지지
않는다. MOM은 사실 데이터를 공급할 뿐, 그 데이터를 어떻게 소비하는지는 하방
서비스의 영역이다.

# **4. MOM AIO 프로토콜 설계**

## **4.1 한 줄 정의**

_UMA처럼 유저가 직접 올리는 오라클 + 근거(언론사 기사) 필수 첨부 + LLM이
자동으로 근거 검증 = 사람이 주장하고, AI가 확인하고, 체인이 기록하는 오라클._

## **4.2 전체 플로우**

**Step 0 (현실 세계): **이벤트 발생. 언론사 기사 발행.

**Step 1 (유저 제출): **Proposer가 claim + evidence URLs + bond 제출. Evidence
최소 1개 필수.

**Step 2 (자동 처리): **(a) Evidence Lite 캡처 - 각 URL의 메타데이터, 콘텐츠
해시, 썸네일 생성. (b) Multi-LLM Proxy 병렬 호출 - GPT-4o, xAI Grok, Gemini,
KoGPT가 독립적으로 '이 증거가 이 클레임을 뒷받침하는가?' 검증.

**Step 3 (온체인 기록): **Evidence + LLM Statement + LLM Provenance(모델ID,
프롬프트해시, 입출력해시) 전부 기와체인에 기록.

**Step 4 (Challenge Period): **72시간 이의제기 기간. 반박 시 Disputer가
counter-evidence + bond 제출 후 동일 파이프라인으로 처리.

**Step 5 (확정): **이의 없으면 자동 확정. 이의 있으면 LLM 비교 판정 또는 MOM
토큰 홀더 투표.

# **5. Evidence Lite - 증거 박제 시스템**

## **5.1 설계 원칙**

Full EoC(전문 저장)는 저작권/삭제권 문제가 있으므로, Evidence Lite 방식을
채택한다. 기사 전문을 저장하지 않고, 메타데이터 + 콘텐츠 해시 + 스크린샷
썸네일만 저장하여 '이 URL에 이 해시의 콘텐츠가 이 시점에 존재했다'를 증명한다.

## **5.2 캡처 대상**

| **항목**                | **저장 방식**      | **저장 위치** |
| ----------------------- | ------------------ | ------------- |
| URL                     | 원본 그대로        | 온체인        |
| 제목                    | 텍스트             | 온체인        |
| 발행사 (publisher)      | 텍스트             | 온체인        |
| 발행일                  | 타임스탬프         | 온체인        |
| 콘텐츠 해시 (SHA-256)   | 해시값             | 온체인        |
| 스크린샷 썸네일         | 이미지 (접힌 형태) | IPFS          |
| 메타데이터 (OG 태그 등) | JSON               | IPFS          |
| 캡처 노드 서명          | EIP-712 서명       | 온체인        |

## **5.3 저작권 안전 설계**

기사 전문을 저장하지 않으므로 저작권법 제136조(복제/전송)에 해당하지 않는다.
메타데이터와 해시값은 저작물이 아니다. 스크린샷 썸네일은 제목+첫 줄만 보이는
접힌 형태로, 인용 범위 내에 해당한다.

## **5.4 삭제권 대응**

Arweave(영구 저장) 대신 IPFS pinning을 사용하여 삭제 요청 시 pin 해제가
가능하다. 온체인 해시는 영구 기록이지만 실제 콘텐츠는 삭제 가능한 저장소에
보관한다.

# **6. Multi-LLM 검증 레이어**

## **6.1 LLM이 검증하는 5가지**

| **검증 항목** | **설명**                        | **예시**                                  |
| ------------- | ------------------------------- | ----------------------------------------- |
| RELEVANCE     | 기사가 해당 claim과 관련 있는가 | BTS 컴백 claim에 BTS 무관 기사 첨부 방지  |
| SUPPORT       | 기사가 claim을 뒷받침하는가     | 기사가 오히려 반대 내용일 수 있음         |
| RECENCY       | 기사 발행일이 적절한가          | 3년 전 기사를 근거로 올리는 것 방지       |
| CONSISTENCY   | 여러 기사 간 내용이 일관적인가  | 기사끼리 모순되면 플래그                  |
| STATEMENT     | 근거 기반 구조화 판정 생성      | YES/NO/AMBIGUOUS + confidence + reasoning |

## **6.2 멀티 모델 병렬 실행**

단일 모델에 의존하지 않기 위해, 동일 Evidence에 대해 4개 이상의 LLM이 독립적으로
판단한다.

| **모델**            | **역할**               | **건당 비용** |
| ------------------- | ---------------------- | ------------- |
| GPT-4o (OpenAI)     | 범용 판단, 영어/한국어 | ~$0.01        |
| xAI Grok            | 타이브레이커, 논리/시간 검증 | 모델별 변동 |
| Gemini (Google)     | 멀티모달, 빠른 처리    | ~$0.005       |
| KoGPT (한국어 특화) | 한국어 맥락 이해       | ~$0.003       |

4개 모델 병렬 호출 시 건당 총 비용: 약 $0.026. 연간 10만 건 처리 시 LLM 비용 약
$2,600.

## **6.3 LLM Provenance (투명성 기록)**

모든 LLM 호출의 입출력이 투명하게 기록된다. UMA의 @OOTruthBot이 블랙박스인 것과
대비하여, MOM AIO는 사후 100% 재현 가능한 구조이다.

| **기록 항목**  | **내용**                                 | **저장 위치** |
| -------------- | ---------------------------------------- | ------------- |
| model_id       | 정확한 모델 버전 (예: gpt-4o-2025-11-20) | 온체인        |
| prompt_hash    | 프롬프트 전문의 SHA-256 해시             | 온체인        |
| input_hash     | LLM 입력 데이터의 SHA-256 해시           | 온체인        |
| output_hash    | LLM 출력 데이터의 SHA-256 해시           | 온체인        |
| full_trace     | 전체 입출력 원문                         | IPFS          |
| prompt_version | 프롬프트 버전 (예: verifier-v1.2.0)      | 온체인        |
| confidence     | LLM 판단 신뢰도 (0-100)                  | 온체인        |

# **7. Publisher Trust Registry**

## **7.1 신뢰 등급 체계**

모든 Evidence의 출처(publisher)에 자동으로 신뢰 가중치가 부여된다. 초기에는
하드코딩, 향후 거버넌스 투표로 변경 가능.

| **Tier**           | **가중치** | **예시**                                            |
| ------------------ | ---------- | --------------------------------------------------- |
| Tier 1 - 최고 신뢰 | 1.0        | 조선일보, 중앙일보, 동아일보, 한겨레, KBS, MBC, SBS |
| Tier 2 - 높은 신뢰 | 0.8        | 연합뉴스, 뉴시스, 이데일리, 머니투데이              |
| Tier 3 - 공식 소스 | 1.0        | DART(공시), 한국은행, 선관위, 기업 공식사이트       |
| Tier 4 - 낮은 신뢰 | 0.3        | SNS (Twitter, Instagram), 커뮤니티                  |
| Unregistered       | 0.1        | 미등록 도메인                                       |

## **7.2 Trust Score 자동 계산**

유저가 제출한 evidence들의 publisher 가중치 평균이 Trust Score가 된다.

| **Trust Score** | **처리 방식**                           |
| --------------- | --------------------------------------- |
| 0.8 이상        | LLM 합의 시 자동 통과 트랙 (fast track) |
| 0.5 ~ 0.8       | 72시간 challenge 기간 필수              |
| 0.5 미만        | 추가 evidence 요구 또는 거부            |

# **8. UMA vs MOM AIO 비교**

| **항목**       | **UMA**            | **MOM AIO**                     |
| -------------- | ------------------ | ------------------------------- |
| 제출 방식      | 유저 직접          | 유저 직접 (동일)                |
| 근거 첨부      | 선택 (거의 없음)   | 필수                            |
| 근거 검증      | 없음               | LLM 자동 검증 (4개 모델)        |
| 근거 보존      | 없음               | Evidence Lite 박제              |
| LLM 활용       | 블랙박스 봇 1개    | 투명한 멀티 모델                |
| 모델 기록      | 없음               | 모델/프롬프트/IO 전부 온체인    |
| 사후 감사      | 불가능             | 완전 가능 (100% 재현)           |
| 분쟁 해결      | UMA 토큰 홀더 투표 | LLM 비교 → MOM 토큰 투표        |
| 출처 신뢰도    | 없음               | Publisher Trust Registry        |
| 보증금         | $750 USDC          | $500 USDC + $0.03 LLM           |
| 속도           | 2시간 ~ 5일        | LLM 수초 + 72시간 대기          |
| 한국 데이터    | 맥락 없음          | 한국어 NLP + Publisher Registry |
| 고래 조작 위험 | HIGH               | LOW (증거 기반 + AI 검증)       |
| 투명성         | LOW (Discord 토론) | HIGH (온체인 전부 기록)         |

MOM AIO는 UMA의 유연한 구조를 유지하면서, UMA의 모든 구조적 약점(근거 부재,
블랙박스, 고래 조작, 사후 감사 불가)을 Evidence + LLM으로 해결한 모델이다.

# **9. 스마트 컨트랙트 아키텍처**

## **9.1 기와체인 (메인 운영)**

| **컨트랙트**            | **핵심 함수**                   | **역할**              |
| ----------------------- | ------------------------------- | --------------------- |
| MOMAIOOracle.sol        | assert(), dispute(), finalize() | 오라클 제출/반박/확정 |
| MOMEvidenceRegistry.sol | registerEvidence()              | Evidence Lite 기록    |
| MOMLLMRegistry.sol      | recordLLMResult()               | LLM Provenance 기록   |
| MOMPublisherTrust.sol   | getTrustScore()                 | Publisher 신뢰 등급   |
| MOMToken.sol (ERC-20)   | transfer(), stake(), vote()     | 거버넌스/스테이킹     |

## **9.2 이더리움 메인넷 (거버넌스 + L1 보안)**

MOM Token(ERC-20)은 이더리움 메인넷에서 발행하고, OP Standard Bridge를 통해
기와체인으로 브리지한다. 거버넌스 투표와 최종 분쟁 해결은 이더리움 L1 보안에
의존한다.

## **9.3 핵심 함수 설계**

assert() 함수: 유저가 eventId + outcome + evidenceUrls + bond를 제출. 백엔드가
Evidence Lite 캡처 + LLM 검증 실행 후 콜백으로 온체인 기록. Evidence 최소 1개
필수, bond 최소 500 USDC.

dispute() 함수: Disputer가 assertionId + counterEvidenceUrls + bond 제출. 동일
파이프라인으로 counter-evidence 처리. 양측 LLM 결과 비교.

finalize() 함수: challenge 기간 종료 후 호출. 이의 없으면 proposer 승. 이의
있으면 LLM 비교 결과 또는 MOM 토큰 홀더 투표로 결정.

# **14. 소셜파이 레이어 — Oracle Contributor Rewards**

## **14.1 리워드 문제의 본질**

예측시장 소셜파이에서 "참여에 따른 리워드"는 핵심 과제다. 돈을 주면 도박이 되고,
안 주면 유저가 안 온다.

| **선택지**                 | **인게이지먼트** | **규제 리스크** | **판정** |
| -------------------------- | ---------------- | --------------- | -------- |
| 현금/토큰 보상 (맞추면 돈) | 매우 높음        | 도박 = 불법     | ❌       |
| 아무 보상 없음             | 매우 낮음        | 없음            | ❌       |
| 환금 불가 포인트           | 중간             | 낮음 (그레이존) | △        |
| **오라클 기여 리워드**     | **높음**         | **없음**        | **✅**   |

## **14.2 프레임 전환 — "예측 보상"이 아니라 "보고 보상"**

핵심은 프레임 전환이다. 유저가 받는 보상은 "미래를 맞춰서" 받는 것이 아니라(=
도박), "이미 발생한 사실을 정확히 보고해서" 받는 것이다(= 노동/서비스 대가).
이는 Amazon Mechanical Turk, 네이버 지식인 채택 답변과 동일한 구조다.

## **14.3 유저 플로우**

**Step 1.** 소셜 피드에서 트렌딩 이벤트 확인 ("BTS 정규 5집 컴백 발표" — 확정
대기 중)

**Step 2.** 유저가 [보고하기] 탭에서 "YES, 컴백 확정" + 조선일보 기사 + 한겨레
기사 첨부

**Step 3.** AIO가 자동 검증 → LLM 4개 모델 합의: "YES, confidence 95%"

**Step 4.** 유저가 받는 것: Oracle Contributor Points(OCP) + "First Reporter"
뱃지(NFT) + Trust Score +5 + 소셜 피드에 "A님이 최초 보고" 표시

## **14.4 OCP(Oracle Contributor Points) 유틸리티**

OCP는 환금 불가이며, 다음 용도로 사용된다:

- **거버넌스 참여:** Publisher Trust Registry 투표, 프롬프트 버전 투표, 새
  이벤트 카테고리 제안
- **프리미엄 기능:** AI 예측 인사이트 열람, 고급 데이터 분석 도구, API 호출권
- **소셜 특권:** 프로필 커스터마이징, 독점 뱃지/NFT, VIP 커뮤니티 접근
- **빌더 할인:** SDK 사용료 할인, 엔진 API 할인
- **(향후) MOM 토큰 에어드롭 자격:** OCP 상위 유저에게 MOM 에어드롭 — 서비스
  기여 보상

## **14.5 소셜파이 게이미피케이션**

- **리더보드:** Trust Score 상위 유저 랭킹
- **뱃지 시스템:** First Reporter, 100연속 정확, K-pop 전문가, K-Finance 전문가
- **프로필:** "이 유저의 보고 정확도 98.5%"
- **소셜 그래프:** 정확한 유저 팔로우 → 알림 → "오라클 인플루언서" 등장
- **댓글/토론:** 이벤트별 근거 공유, 반박 토론

## **14.6 전략적 의미 — 유저 네트워크가 진짜 자산**

SDK + Oracle만으로는 빌더에게 매력이 부족하다. 빌더가 원하는 건 "유저가 있는
생태계"이다.

소셜파이로 유저 10만~100만을 먼저 모으고, "유저 네트워크 + SDK + Oracle"
패키지로 빌더에게 제공하는 것이 핵심 전략이다.

- **Phase 1:** 소셜파이로 유저 모음 (오라클 크라우드소싱 소셜 앱)
- **Phase 2:** 유저 네트워크를 레버리지해서 빌더 온보딩
- **Phase 3:** 합법화 시 빌더 폭발 → 이미 유저 + 오라클 + SDK 다 있는 Traverse가
  유일한 풀스택 인프라

# **15. 예측시장 엔진 모델 — HIP-4 참조**

## **15.1 Hyperliquid HIP-3/HIP-4 모델**

Hyperliquid는 2025~2026년에 HIP-3(Builder-Deployed Perpetuals)과 HIP-4(Outcome
Trading)를 출시하여, 누구나 자기만의 영구선물/예측시장을 Hyperliquid 인프라 위에
배포할 수 있게 했다. 빌더는 오라클, 수수료, 마켓 디자인을 직접 설정하고,
수수료의 50%를 가져간다.

## **15.2 Traverse 적용 — 한국형 예측시장 엔진**

Traverse가 예측시장 엔진(종목 생성 + MOM AIO 오라클 + 정산)을 API/SDK로
제공하고, 빌더가 자기 브랜드로 운영한다.

- **Traverse 제공:** 종목 생성 엔진, MOM AIO 오라클, 정산 시스템, 한국 데이터
  7대 카테고리
- **빌더 결정:** 체인(RPC만 교체), 수수료, UI/UX, 타겟 유저
- **수익 분배:** 거래 수수료의 30~50%를 Traverse가 수취

## **15.3 체인 무관성 (Chain-Agnostic)**

Traverse 엔진은 EVM 호환 체인이라면 빌더가 RPC만 교체하여 어디든 배포 가능하다.
기와체인, Ethereum, Arbitrum, Base, Polygon, BSC 모두 지원. 비EVM 체인은 별도
SDK로 대응한다.

## **15.4 규제 이점**

Traverse = 인프라 제공자 (엔진/API), 빌더 = 마켓 운영자. AWS가 EC2 위에서 누가
무엇을 운영하든 책임지지 않는 것과 동일한 논리로, 규제 리스크가 대부분 빌더에게
이전된다. 단, Traverse는 이용약관에 합법성 빌더 책임을 명시하고, 불법 활동 인지
시 즉시 차단하는 메커니즘을 구비해야 한다.

| K-Data 라이선스        | 한국 7대 카테고리 데이터 공급          | B2B 구독                    |

## **17.3 왜 지금인가**

글로벌 예측시장 $1T 시대가 도래하고, 한국 이벤트가 글로벌 시장에서 거래되고
있으나 신뢰할 오라클이 없다. AI 팩트체크 수요가 폭발하고, 기와체인 메인넷 출시가
임박했다. Traverse의 크롤링 인프라와 한국어 NLP 역량이 이 기회를 포착할 최적의
시점이다.

## **17.4 핵심 메시지**

**Chainlink가 글로벌 가격 표준이라면, MOM은 한국 진실의 표준이 된다. AI +
Evidence + 체인의 결합으로, 세계에서 가장 투명하고 감사 가능한 오라클 인프라를
만든다.**
